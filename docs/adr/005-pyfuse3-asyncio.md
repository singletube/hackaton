# ADR-005: Асинхронная файловая система на pyfuse3

*   **Статус**: Принято
*   **Дата**: 2026-04-17
*   **Контекст**: Для реализации "Гибридного вида" требуется монтирование облака как диска. Обычные FUSE-библиотеки в Python (например, `fusepy`) работают синхронно, что приводит к блокировкам (фризам) интерфейса при задержках сети.

## Рассматриваемые варианты
1.  **`fusepy`**: Просто, но синхронно. Любой тяжелый запрос блокирует весь поток.
2.  **`rclone mount`**: Мощно, но сложно кастомизировать логику "Гибридного вида" под специфические требования ОС (например, ALT Linux).
3.  **`pyfuse3` + `asyncio`**: Современные биндинги к `libfuse 3`, поддерживающие асинхронность.

## Решение
Выбран **`pyfuse3`** с использованием **`asyncio`**.

## Обоснование
1.  **Параллелизм**: Асинхронная обработка системных вызовов (`getattr`, `read`, `readdir`) позволяет системе оставаться отзывчивой даже при медленном интернет-соединении.
2.  **FUSE 3**: Поддержка актуального протокола FUSE 3 обеспечивает лучшую производительность и поддержку современных расширений файловой системы.
3.  **Единый Event Loop**: Позволяет бесшовно объединить логику ФС, сетевые запросы API и работу с БД в одном цикле событий.

## Последствия
- Положительные: Высокая плавность работы интерфейса, отсутствие зависаний проводника.
- Отрицательные: Более высокая сложность кода по сравнению с синхронными аналогами; необходимость тщательной обработки исключений внутри async-функций.
