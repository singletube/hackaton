# Обзор архитектуры CloudBridge

## Концепция
CloudBridge — это гибридная система управления файлами для Linux, которая объединяет локальное хранилище и облако Яндекс.Диск в единый интерфейс. Система построена по принципу "Cloud-First, Local-Mirror", где структура папок всегда синхронизирована, но содержимое файлов загружается по запросу.

## Основные компоненты

### 1. Core (Ядро)
- **Providers (YandexDiskProvider)**: Изолированный слой для работы с внешними API. Генерирует унифицированные события и списки файлов.
- **HybridManager**: Мозг системы. Отвечает за логическое объединение (Union) локального дерева папок и облачного. Решает конфликты имен и путей.
- **StateDB (SQLite)**: "Источник истины" (Single Source of Truth). Хранит текущий статус каждого файла (синхронизирован, в очереди, ошибка).

### 2. Watcher (Фоновый демон)
- Использует подсистему ядра Linux `inotify` (через библиотеку `watchdog`).
- Отслеживает изменения в реальном времени и ставит задачи на синхронизацию в очередь.

### 3. Integration Layer (Nautilus Extension)
- Расширение на Python для проводника GNOME.
- Обеспечивает визуальную связь с пользователем через:
    - **Emblems**: Иконки статуса поверх файлов.
    - **Context Menus**: Действия через правую кнопку мыши.

## Потоки данных (Data Flows)

1.  **Событие пользователя (Drag-and-Drop)**:
    - Файл попадает в папку -> `Watcher` ловит событие -> `StateDB` помечает файл как `syncing` -> `YandexProvider` начинает загрузку -> `StateDB` обновляет статус на `synced`.
    - `Nautilus` видит обновление в БД и меняет иконку с 🔄 на ✅.

2.  **Запуск системы (Discovery)**:
    - `HybridManager` запрашивает дерево у Яндекса -> Сравнивает с БД -> Обновляет локальные "пустышки" (placeholder files).
